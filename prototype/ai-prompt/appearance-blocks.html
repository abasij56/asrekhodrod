طرح Car Blocks، Homepage Blocks و ACF — تم Asre Khodro

← سند جداسازی ظاهر از داده

اصل طلایی: بلوک‌هایی مثل ak-car-specs را دو بار ثبت نکنید (ak-car-specs-classic و ak-car-specs-modern).

محتوا و فیلدهای ACF در post_content ذخیره می‌شوند. اگر slug بلوک عوض شود، با سوییچ ظاهر محتوا می‌شکند.

راه درست: یک بلوک، یک داده، چند ظاهر.

وضعیت فعلی در تم

لایه فایل نقش
Core داده CarAcfFields.php فیلدهای ACF (heading، rows، gallery، ...)
Core ثبت CarBlocks.php acf_register_block_type + render callback
Appearance ظاهر views/blocks/car/*.twig HTML، کلاس‌های CSS، ساختار UI
Core داده + ظاهر Blocks.php بلوک‌های homepage — داده سنگین + Twig

جریان فعلی در CarBlocks::render():

$context['fields'] = get_fields();
\Timber\Timber::render( 'blocks/car/specs.twig', $context );

با سوییچ ظاهر، فقط مسیر Twig و assetها باید عوض شوند؛ فیلد ACF و slug بلوک ثابت می‌مانند.

اصل طراحی: Block = Contract + Appearance Template

┌─────────────────────────────────────────────────────────┐ │ CORE (ثابت) │ │ • CarAcfFields → فیلدها │ │ • BlockRegistry → ثبت ak-car-* یک‌بار │ │ • BlockRenderer → resolve template + build context │ │ • CarBlockData* → (اختیاری) نرمال‌سازی داده │ └─────────────────────────────────────────────────────────┘ │ ▼ ┌─────────────────────────────────────────────────────────┐ │ APPEARANCE (classic / modern) │ │ • views/blocks/car/specs.twig │ │ • assets/css/car-page.css │ │ • assets/js/car-page.js │ └─────────────────────────────────────────────────────────┘

قرارداد بلوک (ثابت بین ظاهرها)

  • slug: ak-car-specs (تغییر نمی‌کند)
  • فیلدهای ACF: section_title, rows, anchor_id, ...
  • context حداقلی: fields, post, block, is_preview

هر ظاهر فقط چگونگی نمایش همان داده را تعیین می‌کند.

جریان Render پیشنهادی

// الان (ثابت)
'template' => 'blocks/car/specs.twig'

// بعد (داینامیک)
Appearance::resolve_block_template( 'car/specs' );
// → appearances/modern/views/blocks/car/specs.twig
// یا fallback → appearances/classic/views/blocks/car/specs.twig

الگوریتم resolve

  1. ظاهر فعال از option خوانده می‌شود (classic / modern)
  2. اگر template در ظاهر فعال وجود داشت → همان استفاده می‌شود
  3. اگر نبود → fallback به classic یا پوشه shared/blocks/
  4. Timber::render() با همان context اجرا می‌شود
// CarBlocks::render() — بعد از refactor
$context  = BlockContext::for_car_block( $block );  // core
$template = Appearance::resolve( 'blocks/car/specs' );
Timber::render( $template, $context );

ثبت ACF و فیلدها در core می‌ماند. فقط مسیر Twig از ظاهر فعال resolve می‌شود.

بلوک‌های Homepage (Blocks.php)

این بلوک‌ها علاوه بر Twig، داده سنگین هم می‌گیرند:

$ctx = Homepage::get_context();  // چندین query
Timber::render( $template, $ctx );

طرح جداگانه برای homepage blocks

  1. در core: HomepageData::section('hero'), ::section('news_list', $count)
  2. هر بلوک در manifest ظاهر اعلام کند چه داده‌ای می‌خواهد
  3. Twig از همان مسیر resolve شود: appearances/{active}/views/blocks/hero.twig
// manifest ظاهر — مثال
'ak-hero' => [
    'template' => 'blocks/hero.twig',
    'data'     => ['hero', 'ticker'],  // فقط این‌ها fetch شوند
],
'ak-news-list' => [
    'template' => 'blocks/news-list.twig',
    'data'     => ['news_list'],
],

ظاهر minimal فقط داده لازم را می‌گیرد → سریع‌تر از Homepage::get_context() کامل.

enqueue: به‌جای Setup با مسیر ثابت، کلاس ظاهر فعال بر اساس is_singular('carsinfo') یا page_has_car_blocks() asset خودش را لود می‌کند.

manifest ظاهر

// appearances/modern/manifest.php
return [
    'id'     => 'modern',
    'blocks' => [
        'ak-car-intro' => [
            'template' => 'blocks/car/intro.twig',
        ],
        'ak-car-specs' => [
            'template' => 'blocks/car/specs.twig',
        ],
        'ak-car-price-chart' => [
            'template' => 'blocks/car/price-chart.twig',
            'script'   => 'assets/js/price-chart.js',
        ],
        'ak-hero' => [
            'template' => 'blocks/hero.twig',
            'data'     => ['hero', 'ticker'],
        ],
    ],
];

BlockRegistry در core از اتحاد manifest همه ظاهرها فقط slugها را برای acf_register_block_type می‌گیرد. template در runtime از ظاهر فعال resolve می‌شود.

تأثیر بر عملکرد

موضوعتأثیر
سوییچ ظاهر روی render بلوک ناچیز (~یک lookup مسیر template)

خطاهای رایج — انجام ندهید

اشتباهچرا بد است
دو slug بلوک per appearance محتوا در DB به slug وابسته است — migration لازم می‌شود
فیلد ACF جدا per appearance داده تکراری و سردرگمی در ادیتور
کپی CarBlocks.php per appearance ثبت دوباره ACF = conflict و خطا
hardcode template path در core سوییچ ظاهر کار نمی‌کند
لود CSS هر دو ظاهر همزمان حجم اضافی — فقط ظاهر فعال enqueue شود

مراحل پیاده‌سازی

فاز ۱ — زیرساخت

  1. کلاس Appearance با resolve() و fallback
  2. انتقال views/blocks/ به appearances/classic/views/blocks/
  3. تغییر CarBlocks::render() برای استفاده از Appearance::resolve()

فاز ۲ — manifest و enqueue

  1. manifest.php برای classic
  2. انتقال car-page.css / car-page.js به پوشه ظاهر
  3. همان کار برای Blocks.php (homepage)

فاز ۳ — ظاهر دوم

  1. ساخت appearances/modern/
  2. فقط بلوک‌هایی که واقعاً متفاوت‌اند Twig جدا بگیرند؛ بقیه از shared یا fallback classic
  3. option active_appearance در ACF
