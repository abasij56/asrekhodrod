Direct answer

2–8 hours is correct for the plugin as it works today — but it is not the physical limit.

With targeted code changes, 30 minutes for 50,000 items is realistic on a decent dedicated server. Default WordPress APIs cannot reach ~28 posts/second when every post triggers attachments, meta, slug sync, and cache purges.

Why the current plugin is slow

Each post goes through class-importer.php → import_one_post(). The work is not one database row — it is a chain of WordPress operations and hooks.

Real cost per post

StepTypical cost
wp_insert_post (large HTML body) ~30–60 ms
save_post_meta — 5–8 × update_post_meta ~20–40 ms
NewsPermalinks::sync_post — extra wp_update_post ~20–40 ms
ExternalMedia::register_url — creates an attachment post ~30–60 ms
set_post_thumbnail + meta ~10–20 ms
Hooks (ACF, LiteSpeed Cache, etc.) Variable — often very large

Total: ~120–250 ms per post in typical conditions.

Where 2–8 hours comes from

  • 50,000 × 150 ms ≈ 125 minutes for posts alone
  • + post categories phase (~50k assignments, batch 25–100)
  • + tags phase (~47k posts with tags)
  • + LiteSpeed “Purge all caches” on many saves → can double or triple total time
Not a mistake in the estimate — it reflects the current implementation. It is also not necessary if the importer is optimized.

What 30 minutes requires

To finish 50,000 items in 30 minutes you need roughly 28 posts per second. WordPress’s normal insert APIs cannot do that on typical hardware. You must bypass the heaviest parts where safe.

Six optimizations (ranked by impact)

1. Don’t create an attachment per post — biggest win

Today the importer creates ~49,991 extra wp_posts rows (attachments) plus featured images. The theme reads _asrekhodro_image_url via ImporterBridge — attachments are not required to display images on the front end.

Action: Skip register_external_thumbnail() during bulk import; save only _asrekhodro_image_url meta.

Effect: Cuts post phase time by ~40–50%.

2. Bulk SQL for posts + postmeta

Replace per-row wp_insert_post with multi-row INSERT where safe, and batch wp_postmeta:

INSERT INTO wp_posts (post_author, post_date, post_date_gmt, post_content, post_title, ...)
VALUES (...), (...), (...);   -- e.g. 1000 rows per query

INSERT INTO wp_postmeta (post_id, meta_key, meta_value)
VALUES (...), (...), (...);

MySQL can handle 5,000–20,000 rows/sec easily. Effect: 5–10× faster than looping wp_insert_post.

3. Bulk taxonomy

Skip wp_set_post_terms per post. Bulk-insert into wp_term_relationships after ensuring terms exist in wp_terms / wp_term_taxonomy.

Effect: Categories + tags in minutes instead of hours.

4. Disable LiteSpeed + heavy hooks during import

wp plugin deactivate litespeed-cache

Re-enable after import completes.

5. Defer slug uniqueness during bulk

Use content-{id} as initial slug; run one SQL pass at the end to rebuild proper slugs from titles if needed.

6. MySQL session tuning (during import only)

SET autocommit=0;
SET unique_checks=0;
SET foreign_key_checks=0;
-- bulk inserts
COMMIT;
SET autocommit=1;
SET unique_checks=1;

Recommended path (safe + fast)

Best balance: Keep wp_insert_post for the main post row (dates, status, slug safety), but remove the expensive surrounding work:

  • No attachment / featured image during bulk
  • Bulk wp_postmeta inserts
  • Bulk wp_term_relationships
  • No cache purges during run
  • Optional overnight pass for attachments or slug polish

Proposed CLI: --fast mode

wp asrekhodro-import run --fast --batch=5000 --taxonomy-batch=500

--fast would enable:

  • Skip attachment / featured image — only _asrekhodro_image_url
  • Bulk INSERT for wp_postmeta
  • Bulk INSERT for wp_term_relationships
  • Suppress LiteSpeed / cache plugin purges during import
  • Optional post-import job for attachments or slug regeneration
