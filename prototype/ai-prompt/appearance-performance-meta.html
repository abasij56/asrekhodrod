عملکرد و متادیتا در refactor ظاهر

بدون meta اضافه per-post — تم Asre Khodro

جداسازی ظاهر از داده معماری بلوک‌ها

پاسخ کوتاه: درست فکر می‌کنید — برای این refactor نباید meta حجیم per-post اضافه کنید.

سوییچ ظاهر یک تنظیم سراسری سایت است (یک مقدار در wp_options)، نه داده‌ای که روی هر یک از ۲۰۰ هزار پست ذخیره شود.

داده کجا ذخیره می‌شود؟

داده محل ذخیره per-post؟
ظاهر فعال (classic / modern) wp_options (ACF Options) خیر — یک مقدار برای کل سایت
فیلدهای car block (rows, gallery, …) همان post (Gutenberg + ACF) — مثل الان بله — refactor تغییری نمی‌دهد
انتخاب Twig / CSS runtime از ظاهر فعال resolve می‌شود خیر — در DB ذخیره نمی‌شود

این refactor نیازی به meta جدید روی هر پست ندارد.

چه کارهایی نباید انجام دهید

anti-pattern چرا بد است
post_meta: _appearance = modern روی هر پست ۲۰۰k ردیف اضافه + query اضافه
فیلد ACF تکراری per appearance روی هر پست ۲ برابر شدن حجم داده ذخیره‌شده
ذخیره HTML رندرشده per post per appearance meta بسیار حجیم
انتخاب template بلوک در meta هر پست غیرضروری وقتی ظاهر global است

هزینه runtime این refactor

در هر request تقریباً:

  1. خواندن active_appearance یک‌بار (با object cache → هزینه ~صفر)
  2. resolve مسیر template: مثلاً appearances/modern/views/blocks/car/specs.twig
  3. همان get_fields() و همان محتوای block مثل الان

نسبت به امروز query اضافه per-post ندارید.

صفحات car الان

داده بلوک‌های car از قبل داخل post است (post_content + داده ACF block). آن حجم به‌خاطر classic/modern بزرگ‌تر نمی‌شود — فقط فایل Twig که همان فیلدها را نمایش می‌دهد عوض می‌شود.

قانون کلی

Global appearance → wp_options (1 row)

Post content → post + existing block data (unchanged)

Per-post appearance → DON'T

اگر همه صفحات از یک skin استفاده می‌کنند، یک option کافی است — بدون meta حجیم per-post و بدون جریمه عملکرد معنادار از خود refactor.

عوامل واقعی عملکرد

تعداد پست در DB تأثیر مستقیم روی render ندارد — مهم تعداد query در هر request است
Page cache مهم‌ترین عامل برای زیر ۱ ثانیه
Object cache options و queryهای تکراری را سریع می‌کند
داده granular homepage فقط sectionهای لازم را بگیرد (بهتر از refactor بد است)
سوییچ ظاهر ~ناچیز — یک lookup + enqueue یک set asset
meta per-post برای ظاهر لازم نیست — avoid

جمع‌بندی

سوالپاسخ
refactor meta حجیم per-post می‌سازد؟ نه — اگر درست پیاده شود
ظاهر کجا ذخیره شود؟ wp_options — یک مقدار سراسری
داده بلوک car تغییر می‌کند؟ نه — فقط template ظاهر
۲۰۰k پست مشکل می‌سازد؟ نه — به‌خاطر تعداد ردیف‌های post_meta ظاهر
چه زمانی per-post لازم است؟ فقط اگر عمداً بخواهید بعضی صفحات skin متفاوت داشته باشند

فرمول نهایی: یک option سراسری + همان محتوای post + resolve template در runtime = بدون meta اضافه، بدون جریمه عملکرد معنادار.
