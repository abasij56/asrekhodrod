جداسازی ظاهر از داده

تحلیل معماری تم Asre Khodro — Appearance Package Pattern

نظر کلی: بله، این ایده کاملاً امکان‌پذیر است و از نظر معماری هم منطقی است — شبیه الگوی «پوسته / Skin» یا «Presentation Package». در وردپرس هم نمونه‌های نزدیک دارد (child theme، Block themes، یا تم‌های چندحالته).

اما جداسازی ۱۰۰٪ تمیز در وردپرس سخت است؛ با این حال می‌توانید به ۸۰–۹۰٪ جداسازی عملی برسید که برای هدف شما کافی است.

وضعیت فعلی تم

الان ساختار تقریباً این‌طور است:

لایه داده (Core) لایه ظاهر (Presentation)
PostTypes, ImporterBridge, NewsPermalinks, CarsInfo views/*.twig
Homepage::get_context(), RfPage, CarbuzzPage assets/css, assets/js
ContactForm, AjaxSearch, PostViews Setup::enqueue_assets()
AcfFields (تنظیمات) Blocks, CarBlocks + Twig بلوک‌ها

جریان فعلی در front-page.php:

$context = array_merge( $context, Homepage::get_context() );
Timber\Timber::render( 'front-page.twig', $context );

یعنی داده از Homepage می‌آید و چیدمان از Twig. این پایه خوبی برای جداسازی است.

نکته: کلاس‌هایی مثل RfPage و CarbuzzPage الان هم داده می‌گیرند هم برای یک layout خاص طراحی شده‌اند — در واقع نیمه‌ظاهر هستند. page-rf.php و page-carbuzz.php عملاً دو «ظاهر جایگزین» هستند که هنوز به‌صورت یک سیستم واحد مدیریت نمی‌شوند.

جریان کار

  1. در ACF Options یک فیلد مثل active_appearance با مقادیر classic / modern
  2. در Theme::init() مقدار را بخوانید (یک بار، با object cache)
  3. کلاس ظاهر فعال:
    • Timber::$dirname را به appearances/{name}/views تنظیم کند
    • فقط CSS/JS همان پوشه را enqueue کند
    • در صورت نیاز template controllerهای خودش را ثبت کند
  4. لایه core فقط ViewModel یا context استاندارد برگرداند

قرارداد بین لایه‌ها

به‌جای اینکه هر ظاهر Homepage::get_context() کامل را بگیرد (که ممکن است ۱۰+ query بزند):

// core
interface SectionDataProvider {
    public function hero(): array;
    public function news_list(int $count): PostQuery;
    public function magazines(int $count): PostQuery;
}

// appearance/classic → فقط hero + news_list + magazines
// appearance/minimal → فقط hero + news_list  ← سریع‌تر!

این جایی است که جداسازی می‌تواند عملکرد را بهتر کند، نه بدتر.

چه چیزهایی کاملاً جدا نمی‌شوند؟

موضوعچرا
add_image_size() در Setupسایز تصویر به URL نهایی وابسته است؛ معمولاً در core می‌ماند
ثبت منوهاداده است، ولی ظاهر تصمیم می‌گیرد کدام منو کجا رندر شود
ACF Blocks در ادیتوراگر هر ظاهر بلوک متفاوت دارد، باید blockها هم per-appearance ثبت شوند
ImporterBridge Twig functionshelper داده‌اند؛ در core می‌مانند
Permalink / Redirectکاملاً مستقل از ظاهر — در core

تأثیر بر عملکرد ۲۰۰ هزار پست هدف < ۱ ثانیه

خبر خوب: تعداد کل پست‌ها در دیتابیس به‌تنهایی سرعت صفحه را تعیین نمی‌کند. مهم است: چند query در هر request، آیا cache دارید، حجم HTML/CSS/JS، و TTFB سرور.

سوییچ ظاهر چقدر هزینه دارد؟

عملیاتهزینه تقریبی
خواندن active_appearance از options~۰.۱ms (با object cache)
لود PHP فقط ظاهر فعالناچیز
enqueue فقط assetهای ظاهر فعالهمان چیزی که الان هم دارید
وجود پوشه دوم روی دیسکصفر — اگر لود نشود

دو پوشه ظاهر روی دیسک، اگر فقط یکی فعال باشد، تقریباً هیچ تأثیری روی runtime ندارد.

چیزهایی که واقعاً سرعت را می‌کشند

الان Homepage::get_context() چندین query پشت‌سرهم دارد (ticker، hero، featured، news_list، magazines، videos، reviews). این مستقل از ظاهر است.

برای هدف زیر ۱ ثانیه، معمولاً لازم است:

  1. Full Page Cache (Redis + Varnish/Nginx FastCGI cache یا WP Rocket)
  2. Object Cache برای queryهای وردپرس
  3. Fragment cache برای بخش‌های homepage
  4. CDN برای media و static assets
  5. هر ظاهر فقط داده‌ای بگیرد که واقعاً نمایش می‌دهد

جداسازی ظاهر اگر درست انجام شود، می‌تواند query اضافی را کم کند؛ ولی بدون page cache، هیچ معماری به‌تنهایی زیر ۱ ثانیه را تضمین نمی‌کند.

مزایا و معایب

مزایا

  • تعویض کل ظاهر با یک option
  • توسعه موازی دو طراحی بدون دست زدن به core
  • تست A/B ظاهر
  • کد تمیزتر و قابل نگهداری‌تر
  • امکان lazy-load داده per-appearance

معایب / ریسک‌ها

  • هزینه نگهداری دوبرابر — هر فیچر UI باید در هر ظاهر پیاده یا shared شود
  • قرارداد context باید stable باشد
  • ACF blocks و page templates نیاز به abstraction دارند
  • migration از ساختار فعلی refactor تدریجی می‌خواهد

توصیه عملی

مرحله ۱ — جداسازی نرم (کم‌ریسک)

  • views/ و assets/ را به appearances/classic/ منتقل کنید
  • یک AppearanceResolver بسازید
  • Setup::enqueue_assets() را به کلاس ظاهر منتقل کنید
  • core دست نخورده بماند

مرحله ۲ — قرارداد داده

  • Homepage را به HomepageData تبدیل کنید با متدهای granular
  • هر ظاهر فقط آنچه لازم دارد را بخواهد

مرحله ۳ — ظاهر دوم

  • وقتی قرارداد stable شد، appearances/modern/ را بسازید
  • option سوییچ را اضافه کنید

جمع‌بندی

امکان‌پذیر است؟ بله — با الگوی Appearance Package
سوییچ با option؟ بله — مسیر Twig + enqueue + controllers
۲۰۰k پست مشکل؟ نه به‌خاطر دو پوشه؛ مشکل از query و نبود cache است
زیر ۱ ثانیه؟ با cache بله؛ بدون cache سخت است
ارزش دارد؟ اگر دو طراحی متفاوت می‌خواهید بله

نقطه طلایی: داده را به ViewModelهای کوچک و مستقل بشکنید و ظاهر فقط مصرف‌کننده آن‌ها باشد — نه اینکه هر ظاهر نسخه کپی‌شده‌ای از Homepage.php داشته باشد.
