بررسی ایده: manifest + zones + repeater تنظیمات + کش — تم Asre Khodro

← معماری بلوک‌ها

جمع‌بندی: ایده جهت درستی دارد و با معماری فعلی تم قابل اتصال است؛ فقط اگر یک‌جا پیاده شود، زود پیچیده می‌شود.

ظاهر = ساختار و جایگاه‌ها (ثابت، در manifest) · تنظیمات = منبع داده و انتخاب بلاک‌ها (متغیر، در ACF) · رندر = layout از کش + query + Twig

وضعیت فعلی تم

در classic/manifest.php ایده شروع شده: pages → sections و blocks تعریف شده‌اند.

ولی front-page.twig هنوز هاردکد است و manifest فقط نقشه است، نه موتور رندر.

HomepageData داده را بخش‌به‌بخش می‌سازد و از ACFهای پراکنده مثل home_main_slider استفاده می‌کند — نه layout repeater واحد.

ایده پیشنهادی، مرحله بعدی طبیعی همین refactor است.

نقاط قوت ایده

۱. تفکیک مسئولیت‌ها

لایهچه کسی تعریف می‌کندمثال
Manifestتوسعه‌دهندهصفحه اصلی → main, sidebar_primary
Options (ACF)مدیر سایتدر sidebar_primary بلاک ak-videos با ۶ ویدیو آخر
RendererCorequery + Twig

این با اصل «یک بلاک، چند ظاهر» در appearance-blocks.html هم‌راستاست.

۲. قرارداد داده عمومی برای بلاک‌های لیستی

برای بلاک‌هایی مثل featured، news-list، videos، magazines:

post_type + category + count + strategy(latest|oldest|manual)

بخشی از این الان هم وجود دارد (featured_category, news_list_count, relationship fields).

۳. کش

layout resolve‌شده را کش کنید (transient/object cache) و فقط هنگام save تنظیمات invalidate کنید. برای homepage خیلی ارزشمند است.

چالش‌های مهم

۱. همه بلاک‌ها «لیست پست» نیستند

  • ak-ticker, ak-hero → اغلب انتخاب دستی
  • ak-newsletter → بدون query
  • content_row_ad → تبلیغات
  • ak-car-* → داده از خود پست

در manifest هر بلاک باید نوع منبع داده داشته باشد:

'ak-videos' => [
  'template' => '...',
  'source'   => 'query',  // query | manual | static | post_meta | ads
  'defaults' => ['post_type' => 'ak_video', 'count' => 6],
],
'ak-newsletter' => [
  'source' => 'static',
],

۲. سایدبار ≠ سکشن اصلی

بهتر است «موقعیت» abstraction واحد باشد — سایدبار فقط یک zone است:

front_page:
  zones:
    before_main: [...]
    main: [...]
    sidebar_primary: [...]
    footer_band: [...]   // مثل picture_frame

۳. UX ریپیتر تو در تو در ACF

«صفحه → موقعیت → بلاک → پست‌تایپ → کتگوری → ...» اگر همه در یک repeater باشد، برای کاربر سنگین است.

پیشنهاد: یک ریپیتر اصلی layout_placements[] — هر ردیف = یک placement با conditional logic بر اساس source بلاک.

۴. محدودیت‌های سازگاری

  • هر zone چه بلاک‌هایی مجازند
  • آیا تکرار مجاز است (ak-hero فقط یک‌بار)
  • ترتیب پیش‌فرض اگر کاربر چیزی تنظیم نکرد

۵. رندر واقعی در Twig

  • front-page.twig فقط حلقه: {% for placement in layout.main %}...
  • یا هر zone یک partial: render_zone('sidebar_primary')

بدون این، manifest فقط مستندات می‌ماند.

پیشنهاد ساختار manifest

return [
  'id' => 'classic',
  'pages' => [
    'front_page' => [
      'label' => 'صفحه اصلی',
      'zones' => [
        'before_main' => [
          'label'   => 'بالای صفحه',
          'blocks'  => ['ak-ticker'],
          'multiple' => false,
        ],
        'main' => [
          'label'   => 'محتوای اصلی',
          'blocks'  => ['ak-hero', 'ak-featured-grid', 'ak-news-list', ...],
          'multiple' => true,
        ],
        'sidebar' => [
          'label'   => 'سایدبار',
          'blocks'  => ['ak-newsletter', 'sidebar-ads', 'sidebar-rail'],
          'multiple' => true,
        ],
        'after_main' => [
          'blocks' => ['ak-picture-frame'],
        ],
      ],
      'defaults' => [
        ['zone' => 'before_main', 'block' => 'ak-ticker'],
        ['zone' => 'main', 'block' => 'ak-hero', 'source' => [...]],
        ...
      ],
    ],
    'archive' => [...],
    'single_post' => [...],
  ],
  'blocks' => [...],
];

defaults خیلی مهم است — وقتی کاربر هنوز چیزی تنظیم نکرده، سایت خالی نماند.

جریان runtime پیشنهادی

1. active_appearance → manifest
2. resolve page (front_page / archive / ...)
3. read ACF placements (یا defaults)
4. normalize + validate + cache
5. برای هر placement:
     BlockDataResolver::resolve(block, config)
6. Twig render با context zoneها

BlockDataResolver جایگزین تدریجی HomepageData::build_* است.

مسیر پیشنهادی (۳ فاز)

  1. فاز ۱: manifest به zones + defaults + رندر داینامیک فقط front_page/main
  2. فاز ۲: ACF repeater placements + BlockDataResolver عمومی
  3. فاز ۳: sidebar، archive، single و بلاک‌های خاص (ads, car, newsletter)
